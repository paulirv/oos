## Spec Document (Principle, Guideline, Success Criterion)

- [ ] Content is clear and unambiguous
- [ ] Uses normative language correctly (MUST, SHOULD, MAY per RFC 2119)
- [ ] Cross-references are valid (links work)
- [ ] Consistent with other spec documents in structure and tone
- [ ] Reviewed for technical accuracy

## Technique Document

- [ ] Includes working code example
- [ ] Example validates (HTML validates, JSON-LD parses)
- [ ] Covers applicable conformance levels
- [ ] Lists benefits and limitations
- [ ] Cross-references relevant success criteria

## Research Document

- [ ] Sources cited or referenced
- [ ] Analysis connects back to OOS design decisions
- [ ] Research questions identified for follow-up
- [ ] Status clearly marked (Draft, In Progress, Complete)

## Reference Implementation / Example

- [ ] Code is syntactically valid
- [ ] Achieves the conformance level it claims
- [ ] Includes comments explaining OOS-specific elements
- [ ] Tested in relevant environment (browser, API client, SSG)
